## 1. 关于 FPGA 验证“线性”关系的理解

### 1.1 今天讨论的核心问题

我们讨论的重点不是：

```text
FPGA 能不能比 NVIDIA GPU 快
```

也不是：

```text
Ozaki 放到 FPGA 上是否一定能超过原生 FP64
```

而是：

```text
低精度模拟 FP64 的软件理论模型，到了真实硬件上是否仍然成立。
```

以 Ozaki 为例，算法理论上把一次 FP64 GEMM 转换成：

```text
k^2 次低精度 GEMM + 高精度重构
```

因此理论上：

```text
effective FP64 throughput ~= low_precision_throughput / k^2
```

或者：

```text
T_measured ~= k^2 * T_low_precision_GEMM
```

这里的“线性关系”指的就是：真实硬件表现是否能被这个 `k^2` 软件模型直接解释。

### 1.2 为什么用 FPGA 做实验

NVIDIA/Rubin 上这个方向能获得较高性能，主要因为 GPU 有很强的低精度 Tensor Core，例如 INT8/FP8 Tensor Core。

但我们想确认：

```text
这是算法本身天然适合硬件？
还是 NVIDIA 在 Tensor Core、cuBLAS、packing、accumulation、dataflow 上做了特殊适配？
```

FPGA 没有 NVIDIA Tensor Core，但它是一个可控实验平台。我们可以用 FPGA 验证：

```text
真实硬件端到端时间
是否接近
k^2 * 低精度 GEMM 时间
```

所以 FPGA 的作用不是复现 NVIDIA 的性能，而是帮助判断软件模型和真实硬件 mapping 的关系。

### 1.3 如果结果是线性的

如果实测接近：

```text
T_measured ~= k^2 * T_low_precision_GEMM
```

说明硬件表现主要由：

```text
低精度 GEMM 吞吐
+ k^2 工作量
```

决定。

这种情况下，研究空间可能比较有限。结论可能变成：

```text
Ozaki 是否值得做，主要取决于硬件的 low_precision / FP64 吞吐比是否大于 k^2。
```

### 1.4 如果结果不是线性的

如果实测明显慢于理论预测：

```text
T_measured >> k^2 * T_low_precision_GEMM
```

说明真实硬件中存在软件模型没有覆盖的瓶颈，例如：

```text
slicing / packing
memory movement
partial result accumulation
reconstruction
on-chip buffer
control overhead
dataflow mismatch
```

这些瓶颈可能就是后续能做出东西的地方。

可能的方向包括：

```text
partial accumulation 留在 device side
slice generation 与 GEMM 融合
减少 partial result movement
改进 reconstruction dataflow
adaptive k / fallback
```

这些优化不一定只适用于 FPGA，也可能帮助解释 NVIDIA 为什么需要特殊硬件或库层适配。

### 1.5 关于 FP8 的补充想法

现在很多新硬件的低精度重点正在从 INT8 转向 FP8/FP4。

因此后续也可以考虑：

```text
FP8 是否能作为 FP64 emulation 的低精度 backend？
```

可以比较：

```text
INT8 slicing:
  FP64 -> k 个 INT8 slice -> k^2 次 INT8 GEMM

FP8-style emulation:
  FP64 -> 多个 FP8 slice / FP8 expansion -> 多次 FP8 GEMM
```

需要关注：

```text
1. FP8 能否达到接近 FP64 的精度？
2. FP8 需要多少 slice / 多少次 GEMM？
3. FP8 的指数位是否有助于处理动态范围？
4. FP8 尾数位较少是否会导致更多 slice？
5. 在有 FP8 Tensor Core 的硬件上，FP8 是否比 INT8 更有潜力？
```

### 1.6 我目前的理解

今天讨论后，我的理解是：

```text
我们用 FPGA 不是为了证明 FPGA 上 Ozaki 更快，
而是为了验证低精度模拟 FP64 的 k^2 理论模型
能否解释真实硬件端到端表现。
```

如果能解释，说明算法和硬件关系比较线性，后续更多是硬件条件判断。

如果不能解释，就需要分析额外瓶颈，并考虑是否能通过数据流、累加、搬运、重构等方式优化。

## 2. 下次组会需要汇报的算法内容

导师还要求下次组会向大家汇报几个核心算法：

```text
DD / FP32 expansion
Ozaki Scheme
ADP-style Ozaki
```

这部分重点不是讲 FPGA 实现，而是先把算法本身讲清楚。

### 2.1 DD / FP32 Expansion

需要说明：

```text
一个 FP64 输入 x 被拆成两个 FP32：
x ~= x_hi + x_lo
```

矩阵乘法可以近似为：

```text
(A_hi + A_lo)(B_hi + B_lo)
= A_hi B_hi
+ A_hi B_lo
+ A_lo B_hi
+ A_lo B_lo
```

常见近似：

```text
DD3: 保留前三项，约 3 次 FP32 GEMM
DD4: 保留四项，约 4 次 FP32 GEMM
```

需要强调：

```text
两个 FP32 不是两个 FP64 的 true double-double。
两个 FP32 只有约 48-bit 有效输入精度，通常达不到 FP64 GEMM 精度。
```

### 2.2 Ozaki Scheme

需要说明：

```text
FP64 输入不是拆成两个 FP32，
而是把尾数 / fixed-point 表示切成多个低精度 integer slice。
```

基本流程：

```text
A -> A_0, A_1, ..., A_{k-1}
B -> B_0, B_1, ..., B_{k-1}

C ~= sum_{i,j} A_i @ B_j * scale_{i,j}
```

计算开销：

```text
k 个 A slices
k 个 B slices
=> k^2 次 INT8 GEMM
```

例如：

```text
k=7 -> 49 次 INT8 GEMM
k=8 -> 64 次 INT8 GEMM
```

需要强调：

```text
Ozaki 的优势来自低精度 Tensor Core 吞吐很高；
代价是 k^2 GEMM、slicing、packing、partial accumulation、FP64 reconstruction。
```

### 2.3 ADP-style Ozaki

ADP-style Ozaki 可以理解为更接近生产实现的 Ozaki 框架。

它不只是固定 k、固定 global scale，而是加入：

```text
exponent span estimation
row / column / block scaling
adaptive k selection
fallback to native FP64
signed / unsigned slice layout
```

需要说明为什么需要这些机制：

```text
global scale 在动态范围很大时会把小元素量化掉；
固定 k 在 easy case 可能浪费，在 hard case 可能不够；
fallback 是为了处理 NaN/Inf、病态输入、大 exponent span 等情况。
```

### 2.4 三者对比

可以用下面的表做汇报：

| 方法 | 核心思想 | 主要开销 | 精度特点 | 硬件依赖 |
|---|---|---:|---|---|
| DD / FP32 expansion | 两个 FP32 表示一个 FP64 输入 | 3/4 次 FP32 GEMM | 明显好于 FP32，但通常达不到 FP64 | 依赖 FP32 GEMM 吞吐 |
| Ozaki | FP64 切成 k 个 INT8 slice | k^2 次 INT8 GEMM | k 足够大时接近 FP64 | 强依赖 INT8 Tensor Core / 低精度吞吐 |
| ADP-style Ozaki | Ozaki + scaling + adaptive k + fallback | 动态变化 | 更鲁棒 | 依赖硬件和库层调度、packing、fallback |

